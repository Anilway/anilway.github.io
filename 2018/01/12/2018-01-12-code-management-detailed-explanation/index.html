1、Git基础知识

什么是git？

分布式版本控制系统，它是由Linux（全球比较大的服务器系统）创始人花两周时间使用C语言编写的工具（在GIT命令窗口中一般都是执行Linux命令）

什么是版本控制系统？

在没有版本控制系统之前，我们的操作可能像下面这样：

创建一个word文档，想要编写自己的年终总结。第一天写了一部分；第二天想接着写，但是发现还要把第一天的部分内容进行修改等，如果直接修改，原有的内容就丢失了，以后也找不到了；

本质问题：后面修改的内容会替换原有的内容，原有的内容没有做保存，以后想找到都找不到 。

版本控制系统的优势：

  1. 备份文件
  2. 记录历史
  3. 回到过去
  4. 多端共享
  5. 团队协作

常用的本本控制系统：

  • Git：分布式版本控制系统
  • svn：集中式版本控制系统

==分布式和集中式的区别==

集中式：

  • 想要做历史记录的查看或者备份，必须链接到中央服务器才可以（需要联网）
  • 处理速度没有Git块

分布式：

  • 每个开发者本地都是一个单独的仓库，在自己的仓库中就可以完成历史版本的记录和查看（不需要联网）
  • git处理的速度更快（git是按照数据存储的）

其余还有一些区别，自己回去扩展。

3、Git工作原理和流程

配置

安装完成git后，我们最好先把一些基础信息配置一下（只需要配置一次即可）。

$ git config -l  查看当前本机git的配置清单

相对比较重要的配置：user.name和user.email。因为每一次提交的时候我们需要知道是谁提交的。

$ git config --global user.name xxx
$ git config --global user.email xxx

设置的用户名和邮箱可以随便写，但是建议最好写github/Codding等平台的账号和邮箱。

工作流程

git是分布式版本控制系统，每一台客户端都是一个独立的git仓库（有git工作的全套机制）。

一个git仓库分为三个区域：

  1. 工作区：平时写代码的地方
  2. 暂存区：把一些写好的代码暂时存储的地方
  3. 历史区：生成一个个版本记录的地方

工作区我们可以看见，暂存区和历史区都以数据的形式保存。

创建Git仓库
$ git init 在当前目录下初始化，把这个目录变成Git可以管理的仓库

在指定目录中执行$ git init，相当于与以当前目录作为基础，创建了一个本地git仓库。

创建完成后，会在项目的根目录中战术.git这个文件（隐藏文件）；有这个.git文件的才叫做git仓库，没有则不能被称为git仓库（因为暂存区和历史区的内容都是存储在这个文件夹中的）。

把工作区的内容提交到暂存区
$ git add xxx 把某个文件提交到暂存区
$ git add .   把所有修改的文件（修改和新增的包含，删除的不包含）提交到暂存区
$ git add -u  把所有修改的文件（包含修改和删除的，但是不包含新增的）提交到暂存区
$ git add -A  是点和u的结合体，所有修改、新增、删除的信息都会提交到暂存区
             （但是点和-A的效果差不多）

$ git status  查看当前文件的状态
              红色：在工作区中，还没有提交到暂存区
              绿色：在暂存区中，还没有提交到历史区

git add . ：他会监控工作区的状态树，使用它会把工作时的所有变化提交到暂存区，包括文件内容修改(modified)以及新文件(new)，但==不包括被删除的==文件。

git add -u ：他仅监控==已经被add的文件==（即tracked file），他会将被修改的文件提交到暂存区。add -u ==不会提交新文件==（untracked file）。（git add –update的缩写）

git add -A ：是上面两个功能的合集（git add –all的缩写）

如果在提交内容的时候，有些内容并不想提交，我们可以增加git提交的忽略文件：.gitignore（没有文件名，只有后缀）

我理解的意思是：如果不想提交某些文件，就把该文件的后缀名、或者文件夹名添加到.gitignore里面。

在提交的时候，就会忽略这些文件，不提交。

// .gitignore中的信息
.idea  // 使用WebStorm打开项目或者编辑项目，自动回生成的文件
node_modules  // 使用npm安装模块的时候，当前安装在项目中的模块都在这个文件夹中（之所以也忽略提交是因为这里的内容太多了）
.DS_Store  // mac本上的文件
...
把暂存区内容提交到历史区
$ git commit
$ git commit -m'版本描述'
$ git commit -a -m'版本描述'   //把提交到暂存区和提交到历史区步骤和在一起完成
                              （这种方式只适合已经提交过至少一次的文件，被修改后，我们可以这个样快速提交。对于新增加的文件，一次都没有提交过，是不允许这样操作的）

//[root-commit]根提交
// 简单理解为，第一次提交到历史区域：如果我们创建一个新的仓库，但是没有做过根提交，此时我们仓库中没有任何的分支（哪怕是master），也就不存在所谓的分支切换（当前仓库还不完整：一个供祖宗流程走完一遍才算完整）

// 我个人理解为，当前仓库还没有进行过任何提交，就不存在历史版本，所以在这种情况下也就不存在分支版本。

// 所以，当你第一次交的时候就是根提交，产生首次历史版本记录。在这之后才能创建分支版本。

查看分支

$ git branch

4、工作流中的一些细节知识

  1. 不管是从工作区提交到暂存区，还是从暂存区提交到历史区，每一个区域当前的内容是一直保存下来的，不会消失。

    个人理解：从工作区提交到暂存区之后，工作区的内容还在。从暂存区提交到历史区后，暂存区的内容也还存在。

  2. 以下两个命令可以查看历史提交记录，也相当于查看历史版本号。在没有历史版本回滚的时候，我们用哪个都可以；有历史版本回滚，git log只能查看当前退回版本以前的版本。

$ git log
$ git reflog
  1. 撤回提交到暂存区
$ git rm --cached . -r
// 可以把 "." 替换为具体的文件名，从暂存区把所有内容（或者是你指定的具体文件）都撤回到工作去。
// 不管暂存区中的内容是否已经提交到历史版本上了，也不管是你第几次放到暂存区的，统统撤回。这种方式太暴力了，我们用的很少

个人理解：

这个撤回命令，会把暂存区中的所有内容统统撤回。相当于清空了暂存区的内容。所以，无论你之前提交了多少次在暂存区，都被撤回了，没有了。只保存了当前工作区的内容。

所以，太暴力，很少用。

  1. 回滚
$ git checkout .

// 把暂存区内容撤回工作区，覆盖现有工作区的内容。而且工作区的内容无法找回。也可以理解为：用上一个暂存区存储的内容覆盖现有的工作区的内容，工作区内容变为和上一个暂存区的内容一样，暂存区内容还在。

// 类似于ctrl+z

个人理解：
回滚的方式，只是用当前暂存区的内容覆盖当前工作区的内容。暂存区的内容还在暂存区。

比如：我现在在工作区写好aaa.txt文件，提交到暂存区。提交后，我继续在工作区对aaa.txt进行添加或者修改。但是我发现还是修改之前的内容好，于是，就像恢复到修改前的样子，也就是刚刚提交到暂存区的时候样子。

那么，就可以通过回滚命令。用当前暂存区的内容覆盖当前工作区的内容。那么，当前工作区的内容就恢复到之前提交的内容。

问题：只能限制当前代码还没有提交的情况，当前代码没提交，回滚的是一次提交到暂存区的内容（和工作区内容不一样）；如果当前这次也提交了，暂存区和工作区一样，回滚回来也是一样的，这个方式就解决不了。

此时需要我们：

  • 在暂存区中，回滚到上一次暂存区中记录的内容（暂存区贤惠滚一次）

    $ git reset HEAD .

// 这个参考文献讲的有些问题，似乎是本次提交到暂存区的内容要提交历史区后。然后下一次如果提交到暂存的内容不满意，才可以用回滚命令回滚到上次暂存区的内容。

  • 把最新暂存区的信息回滚到工作区，替换工作区内容

    $ git checkout .

这些回滚命令都不常用。一般在程序开发过程中，是直接写好一个版本，从共工作区一直提交到历史区的。不会说写一点，存在暂存区一点。

  1. 查看每个区代码区别
$ git diff           // 工作区 VS 暂存区
$ git diff master    // 工作区 VS 历史区
$ git diff --cached  // 暂存区 VS 历史区

查看不同区域之间代码的不同，我们一般都是基于可视化的页面来查看不一样的。

  1. 最重要的回滚
$ git reset --hard 版本号

// git log可以查看版本号。回滚的时候指定的版本号不一定非常全，有个7、8位即可。

当我们回滚到某一个历史版本之后，暂存区和工作区的内容都将被这个版本内容所代替。

  1. 把历史操作输出
history > xxx.txt

5、团队协作开发下的git操作

前面讲的都是单独开发，在自己本地建立git仓库的一系列操作流程，团队协作开发之下，流程上还是有所区别的。

5.1 创建中央仓库

一般是由团队技术leader或者指派的人完成的，仓库中默认是有一些初始化文件的。

中央仓库可能在：gitHub、Coding、自己公司的git仓库服务平台、自己公司的服务器等…。

本教程以gitHub为例。

[1] 基于gitHub创建远程仓库，创建完成后，会生成一个远程地址，例如：https://github.com/xxx/xxxx.git

[2] 作为leader，还需要把项目中一些基础的信息提交到远程仓库上。

  • 在自己本地仓库创建一个仓库，把一些基础内容都放在仓库中
  • 把新增加的信息提交到本地方库历史区中
  • 让本地仓库和远程仓库保持关联
  • 把本地仓库历史区中的信息同步（推送）到远程仓库上

让本地仓库和远程仓库保持关联：

$ git remote add 名字（一般都叫做origin） 远程仓库地址
$ git remote rm 名字 // 移除关联
$ git remote -v      // 查看当前仓库和那些远程仓库保持关联

比如：

$ git remote add test https://github.com/xxx/xxxx.git

就和远程仓库保持关联了。

让本地历史区信息和远程仓库信息保持同步：

$ git push [名字] master // 把本地信息推送到远程上
$ git pull [名字] master // 把远程的拉取到本地

// 这里的[名字]是我们本地和远程仓库关联的那个名字，master是远程仓库的主分支

当然，我们创建完成远程仓库后，可以直接通过克隆仓库地址的方式把远程仓库克隆岛本地：

  • 相当于在本地创建了一个仓库
  • 也让本地这个仓库和远程仓库保持了链接（名字：origin）
  • 也把远程仓库中现有的内容克隆到了本地
  • …
$ git clone 仓库地址 仓库别名 // 仓库别名可以不写，如果不写，就是沿用远程库的名称；

// 使用示例
$ git clone https://github.com/xxx/xxxx.git 测试仓库

5.2 无分支模式下的团队协作

作为开发者每天来的第一件事情或者提交代码之前，都要先pull一下。也就是说不直接把代码提交到中央仓库。先要把中央仓库的文件拉到本地，再进行提交。

因为你的代码可能和同事的代码发生冲突，你需要先在本地从仓库处理好，再进行提交。

如果远程仓库和本地仓库不是同一个文件同一行代码冲突：

  • git会自动依赖于Fast-forward模式进行合并
  • 自动合并后，我们重新提交即可 git add/commit/push

同一个文件的同一行代码冲突

  • 找到冲突的文件，留下自己想要的代码
  • 不管之前是否commit过，都要重新的commit，然后push即可。

5.3 多分支模式下的团队协作

又叫做单独分支管理。

  1. 每天第一件事情，就是创建一个dev分支（也可以叫别的名字），并且切换到这个分支上。
  2. 正常的开发代码，把每天开发的任务都先提交到自己的分支上。
  3. 提交到远程仓库上：
    • 把本地自己分支dev中的内容，合并到本地自己的master分支下
    • 把自己本地创建的分支删除（可以不删除，但是有的公司不希望远程中出现分支，或者避免开发人员的分支冲突，提交之前都要把自己的分支删除掉）
  4. 和第一种只是用master分之一样，把本地最新你合并的master分支代码，提交到远程仓库的master分支下（冲突合并即可）

操作分支的基础命令

$ git branch            // 查看现有的分支
$ git branch [分支名]    // 创建一个新的分支
$ git checkout [分支名]  // 切换到某个分支上（当前切换到某个分支上的时候，会把当前master分支中的信息同步到这个分支上（这个有问题，参考文献没弄清楚））
$ git checkout -b [分支名]   // 创建一个新分支，并且切换到这个分支上
$ git branch -D [分支名]  // 删除某个分支（一定要切换到其他分支上才可以删除当前的分支）
$ git merge [分支名] // 合并分支内容
$ git log --graph / --oneline // 在有分支的情况下，可以更清楚查看分支的提交和合并内容（了解就好）

6、github界面操作

6.1 给别人的仓库提交代码修改或建议

  1. 首先fork别人的仓库
    • 把别人的仓库克隆一份一模一样的，放到自己的账号下，变为自己的仓库（我们平时可以修改自己仓库中的源码）
    • fork的仓库和别人原始仓库会默认建立一些关系，我们可以把自己仓库中和别人仓库中不一样的地方提交给别人（pull-request）
  2. 把自己fork的仓库，克隆到本地
    • 以后有一些新的代码心得，可以自己尝试去修改，然后同步到自己的fork仓库中
  3. 在github中发送new pull request

6.2 github发布非后台项目

github值提供了web站点的发布，后台项目没有提供必要的环境。

把整个仓库作为一个项目发布

这种模式不常用。

master是项目代码，gh-pages分支下存储的是项目预览页面代码

  • 其他人克隆研究的是matser分支下的代码
  • 看介绍页面时gh-pages分支下的代码
  1. 创建一个仓库
  2. 把仓库克隆到本地
  3. 把源码传到master分支上
  4. 在本地仓库中创建gh-pages分支
  5. 把介绍页面放在gh-pages分之下
  6. 把介绍页面的内容上传到github的gh-pages分支下
